## 1. 产品概述

### 1.1 产品愿景

打造一个**无需联网、保护隐私、功能全面**的私人健身伴侣。FitKit 不只是一个动作分析工具，而是一个"健身工具箱"，将所有训练中需要的零散功能（计时、记录、分析、计划）整合于一体，让手机变成随身的智能教练。

### 1.2 产品定位

| 维度 | 说明 |
|------|------|
| 形态 | 纯本地运行的移动应用（iOS / Android） |
| 核心逻辑 | "工具箱"框架，首页陈列多种健身小工具。视频动作分析是最核心、技术含量最高的一个，但不是唯一。 |
| 关键差异 | 完全离线可用，无需注册登录，所有数据与计算均在设备端完成，彻底杜绝隐私焦虑。 |

### 1.3 目标用户

| 用户画像 | 核心痛点 |
|----------|----------|
| 居家健身者 | 无专业教练指导，希望确保动作标准，预防受伤 |
| 隐私敏感人群 | 不愿意将自己的视频、身体数据传输至云端 |
| 健身房新手 | 在健身房使用手机进行动作自检和组间休息管理 |
| 户外/无网络环境训练者 | 常在无稳定网络的地方运动 |

## 2. 核心回答：必须用大模型吗？离线可行吗？

### 2.1 必须使用云端大模型吗？

**不必须。** 视频动作分析的核心能力可以通过端侧轻量级 AI 模型实现，完全不需要云端大模型（如 GPT、Gemini）。原理如下：

1. **人体姿态估计**：使用 MediaPipe Pose、BlazePose 或 MoveNet 等模型，在设备本地从视频流中实时提取 33 个关键身体骨骼点。
2. **动作分类与比对**：将提取的骨骼点序列（角度、轨迹、速度）与预设的标准动作模板进行规则匹配，或使用一个训练好的轻量级分类器（TensorFlow Lite 模型，仅几百 KB），来判断动作是否标准并计数。

### 2.2 做一个完全不用联网的 App 可行吗？

**完全可行，这就是本产品的核心设计原则。** 整个系统架构围绕"离线优先"构建：

- 所有 AI 模型文件在安装时打包进 App
- 所有视频处理、分析均在手机 CPU / GPU / NPU 上完成
- 所有用户数据（训练记录、身体数据）只存储在本地

## 3. 产品功能需求

### 3.1 整体架构：工具箱首页

App 启动后进入工具箱主界面，以可自定义的网格或列表展示所有小工具，用户可自由排序。**视频动作分析只是其中一个工具卡片。**

```
┌──────────────────────────────────┐
│  FitKit - 你的私人健身工具箱      │
│  今日训练: 深蹲 4x15 · 消耗 320kcal │
├────────────┬─────────────────────┤
│ 🎬 视频分析 │  ⏱️ 间隔计时器      │
│ 动作识别   │  自定义循环提醒      │
│ 实时计数   │                     │
├────────────┼─────────────────────┤
│ 📊 训练记录 │  📏 身体测量         │
│ 历史与趋势  │  体重/围度/图表      │
├────────────┼─────────────────────┤
│ 🏋️ 动作库  │  📋 自定义计划       │
│ 标准骨骼演示 │  训练方案组合       │
└────────────┴─────────────────────┘
```

### 3.2 工具集详细功能模块

#### 工具箱（核心框架）

所有小工具的入口与管理中心。

- **"视频动作分析"工具**：卡片上有"开始分析"快捷入口和上次分析摘要。
- **其他必备工具**：

| 工具 | 功能描述 |
|------|----------|
| 间隔计时器 | 自定义训练/休息时间，循环提醒 |
| 训练记录 | 手动或自动记录每次训练的详情，查看历史与趋势 |
| 身体测量 | 记录体重、围度等，生成变化图表 |
| 动作库 | 内置标准动作的 3D 骨骼动画演示与文字要点，是所有工具的数据基础 |
| 自定义计划 | 将不同动作和计时方案组合成训练计划 |

#### 核心工具：视频动作分析（MVP 版功能）

**模式一：实时计数与初筛**

| 阶段 | 说明 |
|------|------|
| 输入 | 调用前置/后置摄像头，实时视频流 |
| 处理 | 在画面中实时绘制人体骨骼点（AR 叠加） |
| 输出 | ① 自动动作计数：识别深蹲、俯卧撑、卷腹等动作的重复次数<br>② 节奏评估：显示每次动作的"向心/离心"阶段耗时，与标准节奏对比<br>③ 基础形态提醒：当关键角度偏差过大时，实时触发视觉（屏幕红色闪烁）和听觉提示。例如："膝盖过度内扣"、"下蹲深度不足" |

**模式二：录制回放分析**

| 阶段 | 说明 |
|------|------|
| 输入 | 从相册导入一段训练视频，或现场录制 |
| 处理 | 全帧扫描分析 |
| 输出 | ① 动作拆解与计数：生成时间轴，标记每一次完整动作的起止点<br>② 逐帧细节回放：可拖动进度条，慢放查看骨骼点重叠对比（理想轨迹 vs 实际轨迹）<br>③ 分析报告：总结性报告，包含总次数、平均速度、最大速度、关键角度达标率等 |

**支持的动作类型（Phase 1）**

- 深蹲（自重/负重）
- 俯卧撑
- 卷腹
- 波比跳
- 硬拉（关键帧检测）
- 平板支撑（计时与髋部位置监测）

## 4. 技术架构设计（完全离线）

### 4.1 架构总览

采用**纯客户端-单机架构**。无服务端，无 API 网关，无数据同步模块。

```
[客户端应用]
├── UI 层 (SwiftUI / Jetpack Compose)
├── 业务逻辑层 (ViewModel / UseCase)
├── 核心引擎层 (Native C++ / Rust)
│   ├── 视频处理模块
│   ├── AI 推理引擎
│   └── 动作规则引擎
└── 存储层 (本地数据库 / 文件系统)
```

### 4.2 关键技术选型

| 模块 | 技术方案 | 用途与说明 |
|------|----------|------------|
| AI 推理框架 | **MediaPipe**（推荐）或 TensorFlow Lite | 提供跨平台、高度优化的姿态估计和动作分类解决方案。MediaPipe 的 Holistic 或 Pose 方案可直接集成。 |
| 姿态估计模型 | MediaPipe Pose Landmarker (BlazePose) | 提供 33 个 3D 身体关键点，速度和准确性在移动端表现极佳，完全离线运行。 |
| 动作分类模型 | 自定义 TFLite 模型 或 基于规则的分类器 | 用于识别深蹲、俯卧撑等具体动作类别。规则分类器（如计算膝盖、髋关节角度）更易维护，无需训练数据。 |
| 视频处理 | FFmpeg (视频解码) / AVFoundation, CameraX | 用于实时相机流捕获、视频文件解码和帧提取。 |
| 前端 UI | **Flutter** / React Native 或原生开发 | 推荐 Flutter：性能优异，一套代码双端运行，且对 C++ 插件（用于集成 MediaPipe）支持良好。 |
| 本地数据库 | Drift (Flutter) 或 Room (Android) / CoreData (iOS) | 存储训练记录、身体测量数据等结构化数据。 |
| 本地存储 | 设备沙盒文件系统 | 存储用户录制的视频、分析报告截图等。 |

### 4.3 核心处理流程（实时计数模式）

```
用户启动分析 (选择动作类型"深蹲"，点击开始)
          │
          ▼
┌─────────────────────────────────┐
│ 1. 相机流接入                     │
│    Camera 模块将 30fps RGBA      │
│    图像数据送入引擎               │
└────────────┬────────────────────┘
             ▼
┌─────────────────────────────────┐
│ 2. 姿态估计                       │
│    AI 推理引擎加载轻量级模型，     │
│    对每帧图像推理，输出 33 个      │
│    关键点的 x,y,z 坐标及可见度    │
└────────────┬────────────────────┘
             ▼
┌─────────────────────────────────┐
│ 3. 状态机追踪                     │
│    动作规则引擎维护状态机，通过    │
│    监测髋、膝、踝的角度阈值变化，   │
│    判断动作状态：                  │
│    预备 → 下蹲 → 底部 → 上升 → 直立 │
│    (= 1 rep)                      │
│    同时检测"膝过脚尖"、"塌腰"等    │
│    异常形态                        │
└────────────┬────────────────────┘
             ▼
┌─────────────────────────────────┐
│ 4. 结果渲染                       │
│    将计数、角度、状态异常等信息     │
│    回传 UI 层，叠加骨骼图和提示     │
│    文字，播放提示音               │
└────────────┬────────────────────┘
             ▼
         循环至暂停，保存训练记录
```

## 5. 非功能需求

### 5.1 性能

| 指标 | 目标值 |
|------|--------|
| 实时分析帧率 | ≥ 25 FPS（近 3 年主流中端手机） |
| 首帧推理延迟 | < 300ms |
| 视频导入分析速度 | ≥ 2 倍速（1 分钟视频 30 秒内分析完） |

### 5.2 资源占用

| 指标 | 目标值 |
|------|--------|
| App 安装包初始大小（含所有模型） | < 80MB |
| 实时分析时内存占用 | < 200MB |
| 使用一小时耗电量 | < 30% |

### 5.3 可用性

- 不需要任何网络权限（iOS 可关闭蜂窝网络和 Wi-Fi）
- 无需任何用户系统（没有注册、登录页面）
- 首次打开 App，在无网络环境下 3 秒内可进入工具箱主页

### 5.4 准确性

| 指标 | 目标值 |
|------|--------|
| 动作计数准确率（良好光照、单一背景、全身可见条件） | ≥ 95% |
| 常见错误形态识别召回率 | ≥ 80% |

## 6. 界面与交互概要

### 6.1 工具箱首页

- 顶部：问候语 + 上次训练摘要（动作、组数、消耗估算）
- 下方：蜂窝状排列的工具卡片，带鲜明图标和简短描述
- 长按卡片可拖拽排序，下拉可添加/移除工具

### 6.2 视频分析主界面

- 全屏相机预览，顶部栏显示所选动作名称和实时计数
- 左下角"模式切换"按钮（实时 / 录制回放）
- 右下角"动作库"入口 + 大号"开始/暂停"按钮
- 分析中：身体骨骼 AR 叠加，颜色表示形态优良/异常
  - 绿色：姿态标准
  - 黄色：轻微偏差
  - 红色：需要修正（配合文字提示）

### 6.3 分析报告页

- 卡片式布局
- 顶部大数字：总次数
- 中部图表：每次动作的耗时分布、关键角度平均得分
- 底部：可滚动的异常标记时间轴，点击可跳转回放对应时刻

## 7. 开发与发布策略

### 7.1 MVP 阶段

- 先实现"视频动作分析"这一个工具，验证离线 AI 引擎的可行性和流畅度
- 支持 1-2 个动作（深蹲、俯卧撑）
- 只做实时计数模式

### 7.2 第二阶段

- 扩展为"健身工具箱"，加入间隔计时器和训练记录
- 累积标准动作库数据，完善规则引擎
- 加入录制回放分析模式
- 支持 6+ 动作类型

### 7.3 第三阶段

- 开放用户自定义动作和规则
- 考虑端侧微调（Federated Learning）以适配个人体型（需充分隐私说明）
- 社区共享动作模板

### 7.4 发布

- App Store / Google Play 上架时必须明确标注**"数据不收集、无需联网"**的隐私标签，这将成为最大获客优势

## 8. 风险与应对

| 风险 | 应对措施 |
|------|----------|
| 端侧模型识别准确率不足 | 初期限制支持的动作类型和使用条件（光照、角度），逐步扩展 |
| 包体积过大 | 按需下载模型文件，基础包仅含核心模型 |
| 不同设备性能差异大 | 设备能力检测后自动切换模型精度（高/中/低） |
| 用户对骨骼点隐私仍存疑虑 | 明确说明所有处理本地完成,提供"不显示骨骼点 AR"的纯文字模式 |

---

> **总结**：FitKit 是一个完全独立、保护隐私且技术可行的产品。核心思路是用极致优化的端侧 AI 小模型替代对云端的依赖，并在产品层包装为功能强大的综合工具箱，在隐私、成本和体验上形成独特竞争力。
